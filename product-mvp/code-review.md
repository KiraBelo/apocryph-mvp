## Критические проблемы (4)

### CRIT-1: SSE работает только на одном сервере
**Файл:** `src/lib/sse.ts`
**Область:** Архитектура / Масштабируемость

**Что это значит простыми словами:** Сейчас уведомления о новых сообщениях (когда партнёр пишет — вы видите сразу) работают через память одного сервера. Это значит:
- Нельзя запустить второй сервер для большей нагрузки
- При перезапуске сервера (деплой) все подключения рвутся
- С ростом пользователей один сервер не справится

**Рекомендация:** Для MVP — задокументировать ограничение. Для роста — заменить на PostgreSQL LISTEN/NOTIFY (уже есть подключение к БД) или Redis pub/sub. Это позволит запускать несколько серверов.

---

### CRIT-2: Куки могут отправляться без шифрования
**Файл:** `src/lib/session.ts:21`
**Область:** Безопасность

**Что это значит простыми словами:** Куки (то, что запоминает что вы залогинены) должны отправляться только по защищённому каналу (HTTPS). Сейчас это зависит от переменной `USE_HTTPS`, и если она не настроена — куки летят по открытому каналу. Злоумышленник в той же сети (кафе, Wi-Fi) может перехватить сессию.

**Рекомендация:** Исправить одной строкой:
```ts
secure: process.env.NODE_ENV === 'production' || process.env.USE_HTTPS === 'true'
```

---

### CRIT-3: Нет защиты от подбора паролей
**Файл:** `src/app/api/auth/login/route.ts`, `register/route.ts`
**Область:** Безопасность

**Что это значит простыми словами:** Злоумышленник может пробовать тысячи паролей в минуту — нет ограничения на количество попыток входа. Также можно массово создавать аккаунты.

**Рекомендация:** Добавить ограничение: 5 неудачных попыток входа с одного IP за 15 минут, 3 регистрации с одного IP в час. Минимум — добавить задержку 500мс при неудачном входе.

---

### CRIT-4: Списки сообщений растут бесконечно
**Файл:** `src/components/hooks/useGameChat.ts:121-127`
**Область:** Масштабируемость (фронтенд)

**Что это значит простыми словами:** Когда партнёр пишет сообщения, они добавляются в список. За долгую игру (часы) этот список растёт бесконечно и никогда не обрезается. Это замедляет браузер и ест память.

**Рекомендация:** Ограничить количество сообщений в списке размером страницы. Когда приходит новое сообщение — либо показывать индикатор «новые сообщения», либо удалять старые с начала.

## Важные проблемы (16)

### IMP-1: Пользовательские URL аватаров/баннеров не проверяются
**Файл:** `src/app/api/games/[id]/route.ts:50` (бэкенд), `MessageBubble.tsx:106` (фронтенд)
**Область:** Безопасность

**Простыми словами:** Пользователь может ввести любой текст как URL аватара — включая вредоносные ссылки (`javascript:`, `data:` URI). Это может использоваться для отслеживания или CSS-инъекций через баннер.

**Рекомендация:** На сервере проверять, что URL начинается с `https://`. На клиенте — базовая валидация перед отправкой.

---

### IMP-2: Нет заголовка Content-Security-Policy
**Файл:** `next.config.ts`
**Область:** Безопасность

**Простыми словами:** CSP — это «фильтр», который говорит браузеру «разрешай скрипты только от нас». Без него, если вдруг пройдёт XSS-атака (вредоносный код в сообщении), браузер выполнит её без вопросов.

**Рекомендация:** Добавить базовый CSP заголовок:
```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' https: data:; connect-src 'self'; frame-ancestors 'none'
```

---

### IMP-3: Нет лимита SSE-подключений на пользователя
**Файл:** `src/app/api/games/[id]/messages/stream/route.ts`
**Область:** Безопасность / Масштабируемость

**Простыми словами:** Один пользователь может открыть неограниченное количество подключений к серверу. Баг в браузере или злонамеренная атака — и сервер падает от нехватки ресурсов.

**Рекомендация:** Ограничить: максимум 5 подключений на пользователя на игру, 20 всего.

---

### IMP-4: Пул подключений к БД слишком маленький
**Файл:** `src/lib/db.ts:10`
**Область:** Масштабируемость

**Простыми словами:** Сейчас максимум 10 одновременных подключений к базе данных. При перезапуске сервера 50+ SSE-подключений пытаются переподключиться одновременно — все 10 заняты, остальные ждут.

**Рекомендация:** Увеличить до `max: 25`, добавить `idleTimeoutMillis: 30000`.

---

### IMP-5: LIKE-инъекция в поиске публичных игр
**Файл:** `src/app/api/public-games/route.ts:38`
**Область:** Безопасность

**Простыми словами:** Пользователь может использовать спецсимволы `%` и `_` в поисковом запросе, чтобы обойти фильтр и найти больше данных чем задумано. Не SQL-инъекция, но нарушает логику поиска.

**Рекомендация:** Экранировать спецсимволы: `q.replace(/[%_\\]/g, '\\$&')` (уже сделано в messages search, но пропущено здесь).

---

### IMP-6: N+1 запросы в ленте публичных игр
**Файл:** `src/app/api/public-games/route.ts:75`
**Область:** Масштабируемость

**Простыми словами:** Для каждой игры в ленте (до 30 штук) отдельно считается количество сообщений. Это 30 дополнительных запросов к базе вместо одного.

**Рекомендация:** Добавить индекс `CREATE INDEX idx_messages_game_type ON messages(game_id, type, created_at)`. Долгосрочно — хранить счётчик сообщений прямо в таблице games.

---

### IMP-7: SSE не проверяет бан после подключения
**Файл:** `src/lib/sse.ts`
**Область:** Безопасность

**Простыми словами:** Если пользователя забанили пока он сидит в игре, он продолжает получать новые сообщения через SSE. Подключение не рвётся.

**Рекомендация:** Раз в 60 секунд проверять сессию в SSE-потоке. Или добавить механизм «выкидывания» при бане.

---

### IMP-8: Нет индекса для фильтрации сообщений по типу
**Файл:** `schema.sql`
**Область:** Масштабируемость

**Простыми словами:** Каждый раз при открытии игры база ищет сообщения определённого типа (IC или OOC). Без правильного индекса это замедляется с ростом количества сообщений.

**Рекомендация:** `CREATE INDEX idx_messages_game_type ON messages(game_id, type, created_at)`

### IMP-9: Кубики используют предсказуемый генератор случайных чисел
**Файл:** `src/app/api/games/[id]/dice/route.ts:27`
**Область:** Безопасность

**Простыми словами:** `Math.random()` не криптографически безопасен. Теоретически можно предсказать результаты бросков.

**Рекомендация:** Заменить на `crypto.randomInt(1, sides + 1)` — одна строка.

### IMP-10: Автоскрытие по 2 жалобам — легко злоупотребить
**Файл:** `src/app/api/games/[id]/report/route.ts:34-44`
**Область:** Безопасность

**Рекомендация:** Поднять порог до 3+, или считать только жалобы от не-участников игры.

### IMP-11: Гонка состояний при обновлении настроек игры
**Файл:** `src/app/api/games/[id]/route.ts:65-73`
**Область:** Целостность данных

**Простыми словами:** Если два игрока одновременно меняют настройки (один — OOC, другой — никнейм), результат может быть непредсказуемым.

**Рекомендация:** Обернуть в транзакцию (`withTransaction`) или использовать `SELECT ... FOR UPDATE`.

**Простыми словами:** Содержимое заметок вставляется как HTML без дополнительной проверки. Хотя сервер чистит HTML, прямое использование `dangerouslySetInnerHTML` — рискованный паттерн.

**Файл:** `src/components/hooks/useGameSearch.ts:47-50`
**Область:** Масштабируемость

**Простыми словами:** При каждом поиске все заметки конвертируются из HTML в текст через создание DOM-элемента. С 50+ заметками это тормозит.

**Рекомендация:** Кешировать текстовые версии заметок при загрузке.

### IMP-16: Нулевая доступность (accessibility)
**Файл:** `src/components/game/*`
**Область:** Доступность

**Простыми словами:** Ни один компонент игрового диалога не имеет ARIA-атрибутов. Люди с ограниченными возможностями (скринридеры, клавиатурная навигация) не смогут пользоваться игрой.

**Рекомендация:** Минимум: `role="dialog"` для модалок, `role="tablist"`/`role="tab"` для вкладок, `aria-live="polite"` для результатов поиска.
